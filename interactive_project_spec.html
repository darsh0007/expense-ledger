Personal Expense & Splitwise Sync Bridge

Master Project Specification & Architectural Outline

1. Executive Summary & Core Concept

This project solves a persistent gap in personal finance software: tracking a strict personal budget limit (such as $800 per month) when your credit cards are regularly used for co-mingled purchases (group groceries, shared tabs, and buying items for others) which are reimbursed asynchronously.

Standard budgeting apps fail because they cannot track running social debts. Splitwise fails because it does not manage personal cash flow or link to personal bank limits. This custom system serves as an intelligent bridge between raw bank transactions and the Splitwise API.

Core Ledger Mechanics: Separating Cash Flow from Expenses

Your application uses a dual-concept ledger system under the hood:

  • Payment Methods (The Cash Flow Source): Your actual credit card accounts or cash wallet. Purchases are logged here at their 100% face value.
  • Asset Accounts (IOUs - The Receivables): This represents "Housemates Owe Me" or "Logan Owes Me." Money assigned here is an asset, not an expense.
  • Expense Accounts (The Consumption - Your $800 Limit): This represents what you actually consumed (e.g., your portion of groceries, your personal coffee). Only these lines subtract from your $800 monthly limit.

When you pay $100 for groceries at Walmart, the full $100 is written to your Credit Card ledger. When you split the transaction on your "free days," $60 is assigned to your personal Grocery Expense (hitting your budget) and $40 is assigned to the "Logan Owe Me" Asset Account. When Logan reimburses you, the $40 shifts from "Logan Owe Me" to your cash/checking account. Your $800 personal limit is never artificially inflated.

3. Database Schema Design (PostgreSQL)

This relational database structure supports multi-party splitting, tracking unprocessed imports, and linking to your outbound Splitwise logs. 

-- Core Accounts representation (Credit cards, cash, and IOU placeholders)
CREATE TABLE accounts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(100) NOT NULL, -- e.g., 'TD Visa Infinite', 'Cash Wallet', 'Roommate Receivables'
    type VARCHAR(50) NOT NULL,   -- e.g., 'credit_card', 'cash', 'asset_iou'
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Master table for raw bank transactions
CREATE TABLE transactions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    transaction_date DATE NOT NULL,
    merchant VARCHAR(255) NOT NULL,
    original_amount DECIMAL(10, 2) NOT NULL, -- The 100% cost as seen on the card
    currency VARCHAR(10) DEFAULT 'CAD',
    payment_method_id UUID REFERENCES accounts(id),
    is_processed BOOLEAN DEFAULT FALSE,     -- Checked once you split the item
    external_bank_id VARCHAR(255),          -- Unique ID from Plaid or CSV to prevent duplicates
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Splits table (A single transaction has 1 or many split rows)
CREATE TABLE transaction_splits (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    transaction_id UUID REFERENCES transactions(id) ON DELETE CASCADE,
    split_type VARCHAR(50) NOT NULL,        -- 'personal_expense', 'roommate_iou'
    assigned_to_user VARCHAR(100) NOT NULL, -- 'Darshan', 'Logan', 'Housemates Group'
    amount DECIMAL(10, 2) NOT NULL,         -- The split portion (e.g., $60.00 or $40.00)
    splitwise_expense_id VARCHAR(100),      -- Tracks corresponding ID in Splitwise
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Optional: Cache Splitwise user/group configurations locally to accelerate your UI
CREATE TABLE splitwise_entities (
    id VARCHAR(100) PRIMARY KEY,            -- Splitwise internal User/Group ID
    name VARCHAR(255) NOT NULL,
    type VARCHAR(50) NOT NULL               -- 'friend' or 'group'
);

5. Phase 1 Execution Details (Getting Started with Sprint 1)

As planned, we are going to start with Sprint 1: The Core Database and CSV Parser. This allows us to get the system fully operational locally before writing any complex third-party authentication code.

Recommended Steps to Initialize Sprint 1:

  1. Initialize PostgreSQL Database: Spin up a free instance using Neon or Railway. Store the database connection string securely in a local `.env` file.
  2. Boilerplate Set Up: Initialize a Node.js Express server with TypeScript.
  3. Create Tables: Run the SQL creation script from Section 3 of this document to set up your tables.
  4. Write the CSV Parser: Write a simple utility using `csv-parser` that takes a standard CSV download from your credit card company and outputs a clean JSON array mapping the Date, Merchant Name, and Amount.
  5. Write Ingest Database Logic: Create a controller function that reads this JSON array and inserts each row into the `transactions` table (setting `is_processed = false` by default). Make sure you check for duplicates so you can upload the same CSV file repeatedly without messing up your ledger database.

6. Cloud Deployment Targets

To keep your project completely free while ensuring a premium developer experience, deploy using the following cloud stack:

  • DB Database (Neon): Serverless Postgres with a highly capable free tier that scales database compute engines to zero when idle.
  • API Backend Server (Railway): Provides automated deployments directly from your GitHub commits, isolated node runtimes, and easy environment variable setup.
  • WEB Frontend (Vercel): Fast static global CDN hosting for your React app dashboard. It auto-deploys on every git commit.
